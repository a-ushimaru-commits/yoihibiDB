# ブランド別内訳 追加設計書

## 背景・目的

既存の「よい日々 売上進捗ダッシュボード」(チャネル別×定期/通常の進捗確認ツール)に、ブランド別(MCTオイル・MSMパウダー・ベルメ等)の内訳を追加する。

きっかけは、ユーザーが別ファイル「1期2期6月比較.xlsx」を提示し「戦略考察シート以外を表示させたい」と依頼したこと。ヒアリングの結果、実際の要件は以下の通り確定した:

- **1期データ**: 固定(変わらない)。既存の1期実績xlsx(`粗利分析_よい日々1期_YYYYMMDD.xlsx`、または同じ`詳細明細`シート構造を持つ「1期2期比較」形式のファイル)から取込む。今回、この`詳細明細`シートの「よい日々」列(ブランド名)を新たに保持する
- **2期月次(チャネル別KPIの元データ)**: 変更なし。既存の`商品別収益YYYYMM.xlsx`(生データ、`商品コード`がFH接頭辞、`金額合計`列を使用 — 直近のバグ修正で確定した仕様)のまま
- **2期日次**: 変更なし。既存の`受注_売上一覧表ライト_YYYYMM.csv`のまま
- **ブランド別データ(新規)**: `分解詳細リスト.xlsx`(商品コード→商品細分の対応表)を新しい取込み対象として追加し、月次実績xlsx・日次CSVの各行の商品コードをこの対応表で引いてブランドを付与する

## なぜ「1期2期6月比較.xlsx」用の専用パーサーが不要か

`1期2期6月比較.xlsx`の`詳細明細`シートは、1期の月(例: `2025-06`)と2期の月(例: `2期 2026-06`)が同じシートに混在している。しかし:

- 既存の`parseBaseWorkbook`は月列を`/^\d{4}-\d{2}$/`で判定しており、`2期 2026-06`のような値は自動的にスキップされる
- つまり、`parseBaseWorkbook`にブランド列を追加するだけで、この種のファイルを1期データとして安全に取込める(2期の行は元々無視される)
- 2期のチャネル別KPIは、実データで数値を実証済みの`商品別収益xlsx`経路を使い続けるため、`詳細明細`シートの2期行を別途パースする必要がない

## データモデル変更

`Record`型にブランドフィールドを追加:

```
Record = {
  yearMonth: "2025-06",
  channel: "TV" | "yahoo" | "その他" | "アマゾン" | "卸" | "楽天" | "自社",
  type: "定期" | "通常",
  brand: string | null,   // 追加。1期データは詳細明細の「よい日々」列から直接取得。
                          // 2期データ(月次/日次)は商品コード→ブランド対応表を引いて付与。
                          // 対応表に無い商品コードは "未分類"
  sales: number,
  cost: number,
  profit: number,
}
```

- 既存の`filterRecords`/`sumRecords`等は`brand`を指定しないフィルタでは無視するため、チャネル別集計(既存機能)には影響しない
- `date`付き日次Recordにも同様に`brand`を追加

## 新規パーサー: 商品コード→ブランド対応表

`分解詳細リスト.xlsx`(シート`Sheet1`、列: 商品コード, 商品名, 商品細分, 定期/単品, 修正)を読み込む:

```
parseBrandLookup(workbook) -> { [商品コード]: ブランド名(商品細分の値) }
```

- ヘッダーは1行目固定(タイトル行なし)。`findHeaderRowIndex`で`['商品コード', '商品細分']`を必須列として検出
- 商品コードをキーとした単純なマッピングオブジェクトを返す
- 空白/重複コードは後勝ち(単純に上書き)で問題ない(473件規模、致命的でない)

## ファイル種別自動判定の拡張

`detectFileType`に新しい戻り値`'brandLookup'`を追加:

- ファイル名が`分解詳細リスト`で始まる、または
- シート`Sheet1`の列に`商品コード`と`商品細分`の両方が存在する

判定の優先順位は既存の`base`/`monthly`/`daily`と同列(ファイル名優先、シート内容フォールバック)。

## ブランド付与ロジック(月次・日次共通)

`parseMonthlyWorkbook`/`parseDailyCsv`に第3引数として`productBrandMapping`(`{商品コード: ブランド名}`)を追加:

```
parseMonthlyWorkbook(workbook, mediaMapping, productBrandMapping)
parseDailyCsv(csvText, mediaMapping, productBrandMapping)
```

各行について:
- 商品コードが`productBrandMapping`に存在する → そのブランド名を使用
- 存在しない(または対応表が空) → ブランド`"未分類"`を割り当て、`unmappedProducts`(戻り値に追加、`unmappedMedia`と同じ形: `{count, sales}`)に集計

集計キーに`brand`を追加: `${yearMonth}|${channel}|${type}|${brand}`。既存のチャネル別集計(brandを見ない集計関数)には影響しない。

## ストレージ変更

`store.js`の状態に`productBrandMapping`を追加(`mediaMapping`と同じ扱い):

```
state = { baseRecords, monthlyRecords, dailyRecords, targets, mediaMapping, productBrandMapping }
```

- `setProductBrandMapping(mapping)`メソッドを追加(`setMediaMapping`と同じパターン)
- 取込み時は`store.getState().productBrandMapping`を各パーサーに渡す

## 未分類ブランドの取込みフロー

ユーザーの要望「該当のブランドがない場合はそのむね表示させブランドの指定後に取込したい」は、既存の未マッピング媒体と同じ「即時取込み＋警告＋設定パネルで修正」方式で実現する(取込みを一時停止してユーザー入力を待つモーダル方式ではない、既存パターンとの一貫性を優先):

1. 取込み時、対応表に無い商品コードの行は自動的に`未分類`ブランドとして集計に含まれる(売上・粗利は失われない)
2. 警告バナーに「未分類の商品コードがあります」と該当コード・件数・売上を一覧表示(`renderMappingWarningsHTML`と同様の`renderBrandWarningsHTML`を新設)
3. 設定パネルに「商品コード→ブランド」の手動割り当て欄を追加。ユーザーが指定すると`productBrandMapping`の上書きとして保存される
4. 上書き保存後、該当月のファイルを再取込みすることで新しいブランド割り当てが反映される(既存の媒体マッピング編集と同じ再取込みルール)

## UI変更

チャネル別テーブルの下に「ブランド別」テーブルを新設:

- 列: ブランド, 売上, 粗利, 粗利率, 1期比(`shiftYearMonth`で1期同月と比較、既存ロジック流用)
- 選択月の`monthlyRecords`にブランドが1件もない場合(すべて生データ取込みで`productBrandMapping`未設定など)は「この月はブランド別データがありません」と表示
- 未分類ブランドの警告バナーをチャネル別警告バナーと並べて表示

## スコープ

- 対象: 月次(商品別収益xlsx)・日次(受注CSV)へのブランド付与、1期データへのブランド保持、ブランド別テーブル表示
- 対象外: ブランド別の月次目標・トレンドグラフ(将来拡張時に追加できる構造にはしておく)
- 対象外: 分解詳細リスト.xlsxの内容自体の編集・追加行対応(取込んだものをそのまま使う)

## エラーハンドリング

- `分解詳細リスト.xlsx`のシート/列が見つからない場合は取込みを中断し、具体的な理由を表示(既存パーサーと同じパターン)
- 商品コードが空欄の行は「未分類」として扱う(クラッシュしない)

## テスト方針

- `parseBrandLookup`の単体テスト(正常系・欠損列エラー)
- `parseMonthlyWorkbook`/`parseDailyCsv`にブランド引数を追加したことによる既存テストの更新(ブランド未指定時は`未分類`になることを確認するテストケースを追加)
- `parseBaseWorkbook`がブランド列を保持することの単体テスト追加
- ブランド別集計・テーブル描画の単体テスト追加
- 実データ(`分解詳細リスト.xlsx`, 既存の月次/日次ファイル)を使ったPlaywright実地検証
